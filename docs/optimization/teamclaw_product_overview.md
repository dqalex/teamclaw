## 一、产品定位

### 一句话定位

> **TeamClaw = AI Agent 管理平台（OpenClaw 的 GUI）**

OpenClaw 是 AI 操作系统（CLI），TeamClaw 是它的系统 UI。TeamClaw **不做 AI**，只做 AI 的**可视化管理、编排和商业化分发**。

### 核心角色

| 角色 | 说明 |
|------|------|
| **运营者（Publisher）** | 内部团队，用 TeamClaw 管理 AI Agent、设计工作流、发布 AI 应用 |
| **消费者（Consumer）** | 外部用户，通过 Marketplace 发现、订阅、使用 AI 应用 |
| **AI Agent** | 由 OpenClaw 驱动的智能体，在 TeamClaw 中被管理、调度和进化 |

### 与竞品的本质区别

| | TeamClaw | Dify / Coze / n8n |
|---|---|---|
| **AI 能力来源** | OpenClaw（外部 AI Runtime） | 自有 AI 引擎 |
| **核心价值** | 管理 + 编排 + 分发 + 变现 | 构建 AI 应用 |
| **消费者分发** | ✅ Marketplace + 激活码 + Credits | ❌ 面向开发者 |
| **Skill 进化** | ✅ 越用越好（经验→规则闭环） | ❌ 静态工具 |

## 二、产品目标

### v1.1 核心目标

| # | 目标 | 衡量标准 |
|---|------|---------|
| **G1** | 多后端适配 | 支持 SQLite / PostgreSQL / MySQL 三种数据库无缝切换 |
| **G2** | Skill 自进化 | Skill 执行质量随使用次数可量化提升 |
| **G3** | 通用工作流 | SOP 升级为支持条件/循环/并行的 Workflow 引擎 |

```
对运营者：描述需求 → Agent 构建 AI App → 自动化宣传分发 → 获得收入

```

## 三、通用协作范式：AI 生成 → 用户干预 → 迭代收敛

**所有场景的底层交互模式一致**。从需求描述到最终交付，中间存在大量由 AI 自动生成的中间产物。由于用户描述不清晰或 AI 理解偏差，一次性生成的结果与预期必然存在 gap。TeamClaw 的解法是：

```
┌────────────────────────────────────────────────────┐
│                                                    │
│  用户描述需求                                       │
│       │                                            │
│       ▼                                            │
│  AI 快速生成中间产物                                 │
│       │                                            │
│       ▼                                            │
│  ┌──────────────────────────────────┐              │
│  │     中间产物可视化展示             │              │
│  │  （用户看到 AI 生成了什么）        │              │
│  └──────────┬───────────────────────┘              │
│             │                                      │
│     ┌───────┼───────┐                              │
│     ▼       ▼       ▼                              │
│   ✅ 确认  ✏️ 手动改  💬 描述给AI改                  │
│     │       │       │                              │
│     │       └───┬───┘                              │
│     │           ▼                                  │
│     │     更新中间产物 ──────▶ 循环迭代              │
│     │                                              │
│     ▼                                              │
│   进入下一阶段（或最终交付）                          │
│                                                    │
└────────────────────────────────────────────────────┘
```

**五条核心设计约束**：

1. **可见性**：AI 生成的每个中间结果必须可视化展示，不允许黑盒
2. **双通道干预**：手动编辑（拖拽/输入）与 AI 对话修改并存，用户自由选择
3. **迭代保护**：AI 重新生成时保留用户已手动修改的部分，不覆盖
4. **阶段逐步收敛**：按阶段逐步推进，每个阶段收敛后再进入下一个，不一次性铺开
5. **随时回退**：任何阶段可回退到之前的阶段重新迭代

> 详见设计原则文档 `原则四：人机迭代协作范式`

## 四、业务场景总览

## 五、各场景详细流程

### S1：项目协作

> 团队成员（人 + AI）在项目中协作完成任务

```
创建团队 → 邀请成员（人 / AI Agent）→ 创建项目
                                          │
                    ┌─────────────────────┼───────────────────┐
                    ▼                     ▼                   ▼
               创建任务             编写 Wiki 文档        提交交付物
                    │                     │                   │
                    ▼                     ▼                   ▼
            分配给成员            双向链接知识管理        审核（通过/驳回）
                    │                                        │
                    ▼                                        ▼
         执行（人工 / AI 自动）                          归档完成
                    │
                    ▼
               标记完成
```

**关键实体**：Team → Project → Task / Wiki / Delivery  
**关键流程**：任务分配 → 执行 → 交付 → 审核

---

### S2：AI Agent 管理

> 运营者注册和管理 AI Agent，控制其行为权限

```
注册 Agent（via OpenClaw）
        │
        ▼
  安装 Skill（从 ClawHub / 本地）
        │
        ▼
  配置信任策略（Trust Policy）
  ├── allow_always：自动执行
  ├── require_approval：需人工审批
  └── deny：禁止操作
        │
        ▼
  分配到项目 → Agent 开始工作
        │
        ▼
  监控运行状态（会话/心跳/快照）
        │
        ▼
  查看效能（Token 消耗、任务完成率、价值产出比）
```

**关键实体**：Agent / Skill / Session / TrustPolicy  
**关键流程**：注册 → 配置 → 分配 → 监控 → 度量

---

### S3：SOP / Workflow 编排

> 运营者设计标准化流程，让 AI 或人按步骤执行

#### 当前（v1.0 SOP 引擎）

```
创建 SOP 模板（定义阶段序列）
        │
        ▼
  阶段 1: input（收集用户输入）
        │
        ▼
  阶段 2: ai_auto（AI 自动执行）
        │
        ▼
  阶段 3: ai_with_confirm（AI 执行 + 人工确认）
        │
        ▼
  阶段 4: render（Content Studio 可视化编辑）
        │
        ▼
  阶段 5: review（提交审核）
        │
        ▼
  阶段 6: export（导出交付物）
```

**7 种阶段类型**：`input` / `ai_auto` / `ai_with_confirm` / `manual` / `render` / `export` / `review`

#### 升级后（v1.1 Workflow 引擎）

```
创建 Workflow（可视化画布拖拽）
        │
        ▼
  节点编排（支持以下类型）：
  ├── sop：调用现有 SOP 模板
  ├── ai_auto：AI 自动执行
  ├── input：等待人工输入
  ├── render：可视化编辑
  ├── review：审核节点
  ├── condition：条件分支（if/else）
  ├── loop：循环节点
  ├── parallel：并行执行
  └── workflow_call：调用子工作流
        │
        ▼
  执行 Workflow（DAG 引擎）
        │
        ├── 支持断点续执行
        ├── 支持运行状态追踪
        └── 向后兼容所有 v1.0 SOP
        │
        ▼
  完成 / 输出产物
```

**关键实体**：Workflow / WorkflowNode / SOPTemplate（兼容层）  
**关键流程**：设计模板 → 编排节点 → 执行 → 断点续执行 → 产出

#### 人机迭代模式（遵循第三章通用范式）

Workflow 编排同样遵循人机迭代协作范式：

| 阶段 | AI 自动生成 | 用户可干预 |
|------|-----------|-----------|
| **节点编排** | 根据需求描述生成节点拓扑 | ✏️ 拖拽调整节点 / 💬 描述修改 |
| **步骤配置** | 自动填充步骤参数和提示词 | ✏️ 修改参数 / 💬 描述调整逻辑 |
| **条件分支** | 自动推荐分支条件 | ✏️ 编辑条件 / 💬 描述调整策略 |
| **测试验证** | 自动生成测试用例 | 确认或补充边界条件 |

### S4：Skill 自进化

> AI Skill 在使用中积累经验，自动提升执行质量

```
Skill 被调用执行
        │
        ├── 自动加载 L1 核心规则（知识库）
        ├── 自动加载 Top 10 历史经验（新增）
        │
        ▼
  执行 Skill 工作流
        │
        ├── 执行正常 → 完成
        │
        └── 用户修正 AI 的输出
                │
                ▼
          自动过滤（跳过一次性错误/纯格式调整）
                │
                ▼
          去重归并（相同场景 → 出现次数 +1）
                │
                ▼
          记录到经验库（L4 层）
                │
                ▼
          检查阈值
          ├── 出现次数 < 3 → 继续积累
          └── 出现次数 ≥ 3 → 建议晋升为 L1 核心规则
                                  │
                                  ▼
                            人工确认晋升
                                  │
                                  ▼
                        写入 L1 核心规则
                                  │
                                  ▼
                    下次执行时自动应用 ✅
```

**知识库五层体系**：

| 层 | 内容 | 更新方式 |
|----|------|---------|
| L1 | 核心规则（≤5 条） | 经验晋升（需人工确认） |
| L2 | 详细标准/规范 | 人工编写 |
| L3 | 案例库/模板 | 人工编写 |
| L4 | 经验记录/踩坑 | AI 自动写入 |
| L5 | 维护日志/变更 | 系统自动记录 |

**关键流程**：执行 → 修正 → 过滤 → 记录 → 积累 → 晋升 → 循环

### S6：主动智能引擎（Proactive Engine）

> 系统主动监控上下文，提前预警和推荐

```
Context Monitor（持续监控项目/任务/交付状态）
        │
        ▼
  Pattern Analyzer（分析模式和异常）
        │
        ▼
  Decision Engine（决策：做什么？优先级？时机？）
        │
        ▼
  Action Executor（执行动作）
  ├── 推送通知
  ├── 建议行动
  └── 自动执行
```

**6 种内置触发规则**：

| 触发场景 | 条件 | 动作 |
|---------|------|------|
| 任务逾期预警 | 到期前 X 小时无人处理 | 推送负责人 + 备选人 |
| 交付积压提醒 | 审核队列 > N 个 | 推送审核者 |
| 上下文断层 | 新任务缺少相关 Wiki | 自动补充文档链接 |
| SOP 建议 | 检测到类似任务模式 | 建议复用 SOP 模板 |
| Skill 健康巡检 | L4 经验超阈值 | 提示归档或晋升 |
| 新人上手 | 新成员加入项目 | 推送项目 Wiki + 快速上手 SOP |

**关键流程**：监控 → 分析 → 决策 → 执行 → 反馈优化

### S7：智能体构建 APP

> Agent 端到端构建 AI 应用，**每个中间产物可见、可编辑、可迭代**，人机协作逐步收敛到满意结果

#### 核心问题 & 解法

从「描述需求」到「应用交付」之间存在大量中间过程。用户描述不清晰或 AI 理解偏差，**一次性生成的结果必然与预期有 gap**。解法不是追求一次完美，而是：

> **AI 快速生成 → 用户看到中间产物 → 手动编辑或描述给 AI 修改 → 迭代直到满意**

#### 分阶段迭代协作流程

```
运营者描述需求（自然语言 / 选模板）
        │
        ▼
  ═══════════════════════════════════
  ║  阶段 1: Workflow 编排           ║
  ║                                 ║
  ║  AI 自动生成:                    ║
  ║  └── Workflow 节点拓扑图          ║
  ║      （AI 步骤/人工步骤/条件分支） ║
  ║                                 ║
  ║  用户干预:                       ║
  ║  ├── ✅ 确认 → 进入下一阶段       ║
  ║  ├── ✏️ 手动编辑（拖拽/修改节点）  ║
  ║  └── 💬 描述修改 → AI 重新生成    ║
  ║                                 ║
  ║  循环直到 Workflow 满意 ✓         ║
  ═══════════════════════════════════
        │
        ▼
  ═══════════════════════════════════
  ║  阶段 2: UI 界面生成             ║
  ║                                 ║
  ║  AI 自动生成:                    ║
  ║  └── Artifact UI 预览            ║
  ║      （输入表单/展示区/交互控件）  ║
  ║                                 ║
  ║  用户干预:                       ║
  ║  ├── ✅ 确认 → 进入下一阶段       ║
  ║  ├── ✏️ 手动编辑（调布局/改样式）  ║
  ║  └── 💬 描述修改 → AI 重新生成    ║
  ║                                 ║
  ║  循环直到 UI 满意 ✓              ║
  ═══════════════════════════════════
        │
        ▼
  ═══════════════════════════════════
  ║  阶段 3: 数据流 & 配置           ║
  ║                                 ║
  ║  AI 自动生成:                    ║
  ║  └── 输入→处理→输出映射           ║
  ║      元数据（名称/描述/定价/分类） ║
  ║                                 ║
  ║  用户干预: 同上三选一              ║
  ║  循环直到配置满意 ✓               ║
  ═══════════════════════════════════
        │
        ▼
  预览 & 测试（沙箱环境）
  ├── 运营者端到端试用
  ├── Agent 自动化测试（边界条件/异常路径）
  └── 发现问题 → 回到对应阶段迭代修改
        │
        ▼
  运营者确认 → 一键发布到 Marketplace（S5）
        │
        ▼
  发布成功 → 触发分发自动化（S8）
```

#### 关键设计约束

> 遵循第三章「通用协作范式」的 5 条设计约束，S7 的具体体现：

| 约束 | S7 具体体现 |
|------|-----------|
| **可见性** | Workflow 拓扑图、UI 预览、数据流配置均可视化展示 |
| **双通道干预** | 拖拽编辑节点/调整 UI 布局 + AI 对话修改并存 |
| **迭代保护** | 用户手动调过的节点/样式，AI 重新生成时保留 |
| **阶段逐步收敛** | 先确认 Workflow → 再确认 UI → 再确认配置 |
| **随时回退** | 任何阶段可回到上一阶段重新迭代 |

#### Agent 构建能力分层

| 层级 | 能力 | 中间产物 | 用户干预程度 |
|------|------|---------|-------------|
| **L1 模板构建** | 基于已有模板微调 | 模板参数配置 | 轻度（改参数） |
| **L2 组合构建** | 组合现有 Skill + Workflow | Workflow + UI + 配置 | 中度（调整编排和 UI） |
| **L3 创意构建** | 从零设计全新 App | 全部中间产物 | 重度（多轮迭代） |

**关键实体**：AIApp / AppWorkflow / AppArtifact / AppSandbox / IterationHistory  
**关键流程**：描述需求 → AI 分阶段生成中间产物 → 用户查看/编辑/描述修改 → 迭代收敛 → 确认发布

### S8：AI 应用分发自动化

> AI 应用发布后，系统自动生成宣传素材并推广，实现零人工的营销分发

#### 发布触发自动化流水线

```
AI App 发布到 Marketplace
        │
        ▼
  ┌─────────────────────────────────────────┐
  │           分发自动化引擎                   │
  │                                         │
```
