## 二、Agent 交互点深度盘点

### 2.1 交互点定义

根据设计原则，交互点是 **Agent 与系统发生信息交换的关键节点**。每个交互点应：
1. 明确上下文传递内容
2. 符合最近前置原则
3. 保护 LLM 注意力
4. 形成闭环

### 2.1a 交互点三层架构

```
┌─────────────────────────────────────────────────────────────────┐
│  Layer 1: 系统级交互点                                            │
│  - .teamclaw-index (Workspace 索引)                              │
│  - 心跳状态、文件索引、同步模式                                     │
│  - Agent 读取 Workspace 状态的入口                                │
├─────────────────────────────────────────────────────────────────┤
│  Layer 2: 业务级交互点                                            │
│  - 推送模板 (task-push, sop-task-push, delivery-push...)         │
│  - Chat 上下文 (chat-task, chat-schedule)                        │
│  - 确认请求 (sop-confirm-request)                                │
│  - 结果通知 (sop-stage-result, delivery-review-result)           │
├─────────────────────────────────────────────────────────────────┤
│  Layer 3: 能力级交互点                                            │
│  - SKILL.md (Skill 定义文件)                                     │
│  - Agent 能力激活和执行的入口                                      │
│  - 业务逻辑封装层                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 上下文注入问题

**问题 1：L4 经验记录未在推送时注入**

```
现状：
- 推送时只注入 L1（项目/里程碑关键信息）
- L4 经验记录存储在知识库中，但推送时不读取
- Agent 执行后沉淀到 L4，但下次推送不读取

问题：
- 经验无法复用
- 同一错误可能重复发生
- 不符合"下次推送自动读取"的闭环设计
```

**问题 2：推送模板过于冗长（破坏注意力）**

```
行数统计：
- task-push.md: 139 行
- batch-task-push.md: 100 行
- sop-task-push.md: 214 行

问题：
- 执行流程说明过长
- 工具使用指南重复
- Agent 注意力分散
```

**问题 3：Chat 上下文静态注入**

```
现状：
- Chat 开始时注入一次任务/定时任务信息
- 执行过程中无动态上下文更新

问题：
- 长时间对话后上下文过时
- 任务状态变更后 Chat 中不知晓
```

## 四、闭环完整性检查

### 4.1 闭环定义

完整的闭环应包含：
1. **消费**：Agent 读取上下文执行任务
2. **生产**：Agent 产出结果/经验
3. **沉淀**：结果/经验保存到知识库
4. **复用**：下次任务自动读取沉淀内容

### 4.2 各业务流程闭环评估

#### ✅ 闭环完整：普通任务执行

```
[用户创建任务] → [task_push 注入 L1] → [Agent 执行]
                                              ↓
[下次推送读取 L4] ← [知识库存储] ← [完成时提示沉淀]
```

| 环节 | 状态 | 说明 |
|------|------|------|
| 消费 | ✅ | 推送时注入 L1 |
| 生产 | ✅ | Agent 执行任务 |
| 沉淀 | ⚠️ | 有提示，但依赖 Agent 主动 |
| 复用 | ❌ | 推送时不读取 L4 |

#### ✅ 闭环完整：SOP 阶段推进

```
[sop_task_push] → [Agent 执行阶段] → [advance_sop_stage]
       ↑                                         ↓
[推送下一阶段] ← [阶段完成通知] ← [保存阶段产出]
```

| 环节 | 状态 | 说明 |
|------|------|------|
| 消费 | ✅ | 推送时注入 L1-L3+前序产出 |
| 生产 | ✅ | Agent 完成阶段 |
| 沉淀 | ✅ | 阶段产出保存到 stageHistory |
| 复用 | ✅ | 下一阶段推送时读取前序产出 |

#### ✅ 闭环完整：SOP 确认流程

```
[Agent 完成阶段] → [request_sop_confirm] → [人工确认界面]
       ↑                                          ↓
[继续下一阶段] ← [确认结果通知] ← [人工点击确认/驳回]
```

| 环节 | 状态 | 说明 |
|------|------|------|
| 消费 | ✅ | Agent 读取上下文 |
| 生产 | ✅ | Agent 产出待确认内容 |
| 沉淀 | ✅ | 确认结果保存 |
| 复用 | ✅ | 确认后推进或重做 |

### 6.6 .teamclaw-index 交互点分析

**当前设计：**
```yaml
version: "1.0.0"
workspace_id: "ws_xxx"
heartbeat:
  status: active
  last_heartbeat: "2026-03-17T10:00:00Z"
files:
  path/to/file.md:
    id: "doc_xxx"
    hash: "abc123"
    version: 3
```

**问题：**
1. **Agent 只读不写**：Agent 无法更新索引（如检测到文件变更）
2. **无变更通知**：文件变更后无事件通知 Agent
3. **无冲突解决提示**：version 冲突时无解决指南
4. **无文件状态**：是否正在编辑、是否有未提交变更等状态缺失
5. **索引过大**：大型 workspace 文件多时，索引文件过大

**建议：**
```yaml
version: "1.1.0"
workspace_id: "ws_xxx"
heartbeat:
  status: active
  last_heartbeat: "2026-03-17T10:00:00Z"
  # 新增：Agent 可报告自己的状态
  agent_status:
    agent_id: "agent_xxx"
    current_task: "task_xxx"
    status: "working"
files:
  path/to/file.md:
    id: "doc_xxx"
    hash: "abc123"
    version: 3
    # 新增：文件状态
    status: "synced"  # synced | modified | conflict | locked
    last_modified_by: "agent_xxx"
    last_modified_at: "2026-03-17T10:00:00Z"
    # 新增：变更提示
    change_hint: "Agent 正在编辑此文件"
# 新增：变更事件队列（供 Agent 消费）
events:
  - type: "file_modified"
    file: "path/to/file.md"
    timestamp: "2026-03-17T10:00:00Z"
    by: "agent_xxx"
```

## 附录 A：三层交互点对比分析

### A.1 各层交互点特征

| 特征 | 系统级 (.teamclaw-index) | 业务级 (推送模板) | 能力级 (SKILL.md) |
|------|--------------------------|-------------------|-------------------|
| **触发频率** | 高（心跳 120s） | 中（任务推送） | 低（关键词触发） |
| **内容大小** | 中（YAML 文件） | 大（模板 100-200 行） | 大（Skill 文档） |
| **Agent 参与度** | 被动读取 | 被动接收 + 主动执行 | 主动激活 + 执行 |
| **闭环完整性** | ✅ 双向同步 | ⚠️ 部分闭环 | ❌ 无闭环 |
| **版本控制** | ✅ 有 version 字段 | ❌ 无版本 | ❌ 无版本 |
| **经验沉淀** | ✅ 文件变更历史 | ⚠️ 需 Agent 主动 | ❌ 无沉淀机制 |

### A.2 交互点设计原则（补充）

#### 系统级交互点设计原则

1. **轻量化读取**：索引文件不宜过大，大型 workspace 应分页
2. **事件驱动变更**：文件变更通过事件队列通知，而非全量读取
3. **双向状态同步**：Agent 应能报告自己的状态到索引

#### 能力级交互点设计原则

1. **版本控制**：SKILL.md 必须有版本字段，支持更新感知
2. **前置条件检查**：执行前检查环境、工具、文件等前置条件
3. **执行跟踪**：记录 Skill 执行历史，支持效果评估
4. **反馈闭环**：Skill 执行后收集反馈，持续改进
